### [fix] Scope quiz submit to the authenticated caller (spec 0005 token-scoping gap)
- **Date:** 2026-07-15
- **Area:** backend (QuizService)
- **What:** Closed the **"Spotted adjacent, not fixed"** token-scoping gap flagged in the submit-idempotency entry directly below. `POST /api/attempts/{attemptId}/submit` → `TakeQuizFacade.SubmitQuizAsync` loaded the attempt by id and submitted with **no ownership check** — unlike `StartQuiz`/`GetResult`, `SubmitQuiz` never read `GetCurrentUserId()`, so any authenticated student could submit answers to another student's attempt by knowing/guessing its id. Violated the token-scoping non-negotiable (README: "tenant scope comes from the token, not the client"), code-standards §5, and security.md §4/§7. Latent today (no take-quiz UI yet) but reachable once the take-quiz UI — the next 0004 child — ships.
  - **Fix:** thread the caller's `Guid studentId` from the controller (`GetCurrentUserId()`, read **before** the try like the sibling endpoints, so a bad identity is 401 not a swallowed 400) into `SubmitQuizAsync`. The facade now loads the attempt **up front** and returns `null` when it is missing **OR** `attempt.StudentId != studentId` — the exact `GetResultAsync` shape — so the controller answers **404 either way** and never reveals that another student's attempt exists (mirrors AC-9). (3) **Infrastructure:** `RefreshTokenGenerator` (32 random bytes → base64url; stored as SHA-256 hex), `RefreshTokenRepository` (single-transaction `TryRotateAsync`, set-based `RevokeFamilyAsync` via `ExecuteUpdateAsync`), `AuthDbContext` + `AddRefreshTokens` migration **applied to live authdb**. (4) **API:** `POST /api/auth/refresh` + `POST /api/auth/logout`; the controller owns *all* cookie I/O (`HttpOnly`, `SameSite=Lax`, `Path=/api/auth`, host-only, `Secure` from config). `JwtTokenService`'s hardcoded `AddHours(8)` → `AuthTokens:AccessTokenMinutes` (default 15). - **Result:** build 0 errors; **tests 42/42** (27 new). Live end-to-end run against real Postgres passed **every acceptance criterion AC-1..AC-5**: cookie flags correct, access token exp = 899s, rotation replaces the cookie, a replay inside the grace window returns 200 with **no** `Set-Cookie`, the same replay after 11s returns 401 **and revokes the whole family** (successor also 401), logout → 204 → refresh 401, logout idempotent. DB check: all `TokenHash` rows are 64-char hex; no raw token is persisted.
- **Notes:** ⚠️ **Spec bug found and worked around — `/architect` must fix AC-3.** AC-3 says a token replaced inside the grace window "returns its successor". That is **unbuildable**: the spec's own Key invariant says the raw token is never stored, and only its SHA-256 hash exists server-side, so the successor's raw value cannot be produced 200ms later. Implemented the correct equivalent instead: **inside the grace window mint a new access token, do not rotate, and do not touch the cookie** — the browser already holds the successor cookie from the winning tab's response. Same intent, same security, invariant intact. Run `/architect 0001: AC-3's grace path contradicts the raw-token invariant; reword to "mints a new access token without rotating or re-setting the cookie"` to backfill. Also: the xmin concurrency token means a lost rotation race is caught and routed through the same benign-replay rule rather than minting a second live token in the family. Confirmed (as in QuizService) that Npgsql omits `xmin` from the generated table — it stays a system column.
