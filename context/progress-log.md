### [feature] AuthService refresh + logout — rotating refresh tokens (spec 0001, build-plan task 1)
- **Date:** 2026-07-10
- **Area:** backend (AuthService) / db / security
- **What:** Built the backend prerequisite for the SPA session model. (1) **Domain:** `RefreshToken` entity (rich, private setters, private EF ctor) with `SessionId` families, `ReplaceWith`, `Revoke`, `WasReplacedWithin`; `IRefreshTokenRepository`; `IAuthUserRepository.GetByIdAsync`. (2) **Application:** `RefreshTokenService` (issue / rotate / reuse-detect / revoke), `AuthTokenOptions` (bound from the new `AuthTokens` config section), `AuthResult` / `RefreshOutcome` / `IssuedRefreshToken` models; `AuthAppService` now opens a fresh session family per sign-in. (3) **Infrastructure:** `RefreshTokenGenerator` (32 random bytes → base64url; stored as SHA-256 hex), `RefreshTokenRepository` (single-transaction `TryRotateAsync`, set-based `RevokeFamilyAsync` via `ExecuteUpdateAsync`), `AuthDbContext` + `AddRefreshTokens` migration **applied to live authdb**. (4) **API:** `POST /api/auth/refresh` + `POST /api/auth/logout`; the controller owns *all* cookie I/O (`HttpOnly`, `SameSite=Lax`, `Path=/api/auth`, host-only, `Secure` from config). `JwtTokenService`'s hardcoded `AddHours(8)` → `AuthTokens:AccessTokenMinutes` (default 15). (5) **New `AuthService.Tests`** project (xUnit + Moq, added to `QuizApp.sln`): 27 tests.
- **Result:** build 0 errors; **tests 42/42** (27 new). Live end-to-end run against real Postgres passed **every acceptance criterion AC-1..AC-5**: cookie flags correct, access token exp = 899s, rotation replaces the cookie, a replay inside the grace window returns 200 with **no** `Set-Cookie`, the same replay after 11s returns 401 **and revokes the whole family** (successor also 401), logout → 204 → refresh 401, logout idempotent. DB check: all `TokenHash` rows are 64-char hex; no raw token is persisted.
- **Notes:** ⚠️ **Spec bug found and worked around — `/architect` must fix AC-3.** AC-3 says a token replaced inside the grace window "returns its successor". That is **unbuildable**: the spec's own Key invariant says the raw token is never stored, and only its SHA-256 hash exists server-side, so the successor's raw value cannot be produced 200ms later. Implemented the correct equivalent instead: **inside the grace window mint a new access token, do not rotate, and do not touch the cookie** — the browser already holds the successor cookie from the winning tab's response. Same intent, same security, invariant intact. Run `/architect 0001: AC-3's grace path contradicts the raw-token invariant; reword to "mints a new access token without rotating or re-setting the cookie"` to backfill. Also: the xmin concurrency token means a lost rotation race is caught and routed through the same benign-replay rule rather than minting a second live token in the family. Confirmed (as in QuizService) that Npgsql omits `xmin` from the generated table — it stays a system column. Migration applied with an explicit `ConnectionStrings__DefaultConnection` override because `appsettings.Development.json` points at the docker host `postgres`, not `localhost`. **Still unwired: no auth middleware and no CORS in AuthService** — correct for now, the SPA reaches it same-origin through the Vite proxy (foundation §7 #27).
